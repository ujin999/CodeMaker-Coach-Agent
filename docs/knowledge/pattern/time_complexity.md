# Time Complexity Guidance (시간 복잡도 분석 패턴)

- **Korean Keywords**: 시간 초과, 시간 복잡도, 제한 시간, 데이터 크기, Big-O 표기법
- **English Keywords**: Time Limit Exceeded, TLE, Time Complexity, Big-O, Input Size, Algorithm Selection

## Concept Summary
코딩 테스트 채점 환경에서 주어지는 연산 시간 제한(보통 1~2초) 내에 통과하기 위해서는 문제 조건에 주어진 입력 데이터의 최대 크기($N$)를 보고 허용 가능한 가장 최적의 알고리즘 시간 복잡도를 유추해야 합니다.

## When to Use
- 오답 사유 중 "시간 초과(TLE)" 상태가 발생하여 작성한 코드의 연산 효율성을 대폭 개선해야 할 때.
- 문제를 설계하고 생성할 때 특정 난이도와 시간 제약 수준에 최적화된 자료구조와 논리를 강제하고자 할 때.

## Input Size and Maximum Complexity Rules
- $N \le 10$: $O(N!)$ 또는 $O(2^N)$ 완전 탐색, 순열, 조합 가능.
- $N \le 100$: $O(N^3)$ 플로이드-와샬 등 삼중 반복 연산 가능.
- $N \le 2,000$: $O(N^2)$ 이중 반복 루프를 사용한 정렬, 다이나믹 프로그래밍 등 가능.
- $N \le 100,000$: $O(N \log N)$ 이진 탐색, 힙 정렬, 병합 정렬 등 가능.
- $N \le 10,000,000$: $O(N)$ 선형 스캔, 누적 합, 투 포인터, 그리디, 단순 해시 테이블 연산 가능.
- $N > 10,000,000$: $O(\log N)$ 또는 $O(1)$ 이분 탐색, 수학적 단순 수식 계산 필요.

## Common Mistakes
- 입력값의 최댓값 연산을 어림잡지 못하고 무작정 $O(N^2)$ 이중 루프로 코드를 짜기 시작해 TLE를 유발하는 경우.
- 재귀 함수 사용 시 하위 문제들의 중복 연산에 캐시 처리를 안 해 지수 시간 복잡도로 폭증하는 경우.
- Python 등 스크립트 언어의 내장 리스트 슬라이싱(`list[a:b]`)이나 검색(`item in list`)의 내부 시간 복잡도($O(N)$)를 무시하고 반복문 내에 혼용하는 경우.

## Problem Generation Guidance
- 문제의 난이도가 올라갈수록 데이터의 한계치($N$)를 크게 늘려 정답 모델이 단순 Brute-Force 대신 세련된 시간 최적화 기법을 쓰도록 유도하십시오.
- 시간 제한을 "1초"로 고정하고, 입력 크기 $N$을 난이도별 가이드라인에 상응하게 조절하여 배치하십시오.

## Testcase Generation Guidance
- 시간 복잡도가 타이트한 $O(N \log N)$ 인지 $O(N^2)$ 인지 판별하기 위해 최대 입력 데이터($N = N_{max}$)를 꽉 채운 극한의 시간 테스트케이스를 필수적으로 포함해야 합니다.
- 정렬 기준에 민감한 저격 케이스(예: 이미 역순으로 정렬된 입력 등)를 포함하여 비효율적인 최악의 시나리오가 걸러지게 하십시오.
