# 요구사항 명세 (Requirements)

CodeMaker Coach Agent의 기능/비기능 요구사항을 정의한다.
기획 원문(`주제 선정.pdf`)의 MVP 범위 + 채점 웹 서비스 + 코드 공유 기능을 반영한다.

- 표기: **MUST**(필수) / **SHOULD**(권장) / **MAY**(선택)
- MVP 범위는 `[MVP]`로 표시한다.

---

## 1. 대상 사용자

- 코딩테스트를 준비하는 대학생 / 알고리즘 초·중급 학습자
- 취업·이직 준비 개발자 지망생
- 특정 알고리즘 유형을 반복 학습하려는 사용자

---

## 2. 기능 요구사항 (Functional)

### 2.1 문제 생성 `[MVP]`
- **FR-1 (MUST)** 사용자는 알고리즘 유형·난이도·문제 스타일·언어·힌트 방식을 선택할 수 있어야 한다.
- **FR-2 (MUST)** 시스템은 선택 조건에 맞는 **새로운** 코딩테스트 문제를 생성해야 한다. (기존 문제 복제 금지)
- **FR-3 (MUST)** 생성 결과는 제목·난이도·알고리즘·학습목표·문제설명·입출력형식·제한조건·예제입출력·
  reference solution·시간복잡도·**단계별 힌트(1~3단계)**를 포함해야 한다. (기획 9.2 JSON 스키마 준수)
- **FR-4 (MUST)** 문제 생성 전 RAG로 관련 알고리즘 개념 문서를 검색해 생성 근거로 활용해야 한다.
- **FR-5 (MUST)** 문제를 생성할 때 **힌트 정보를 함께 생성**하여 DB(및 힌트용 벡터 스토어)에 저장해야 한다.
  힌트는 문제와 1:N으로 연결되며, 각 힌트는 다음을 포함한다.
  - `level` (1=방향, 2=알고리즘 접근, 3=구현 포인트)
  - `content` (해당 단계에서 노출 가능한 설명)
  - `reveals_core_code = false` (원칙적으로 핵심 코드는 담지 않는다 — FR-19 참조)
  - (선택) `code_skeleton` (핵심 로직을 비운 구조/뼈대 코드)
  > 힌트는 풀이 시점에 LLM이 즉석 생성하는 것이 아니라, **문제 생성 시 미리 만들어 저장한 힌트를 RAG로 서빙**하는 것을 기본으로 한다.
  > (일관성 확보 + 정책 통제 + 응답 속도/비용 절감 목적)

### 2.2 문제 검증 `[MVP]`
- **FR-6 (MUST)** Testcase Generator가 sample / hidden / edge 테스트케이스를 생성해야 한다.
- **FR-7 (MUST)** Reference Solver가 내부 정답 코드를 생성해야 한다.
- **FR-8 (MUST)** Validator가 예제 출력 == reference solution 실행 결과 일치 여부, 조건 일관성,
  난이도 적합성, 시간복잡도 적합성을 검증해야 한다.
- **FR-9 (MUST)** 검증 실패 시 문제를 **재생성**해야 한다. (LangGraph 조건 분기)
- **FR-10 (SHOULD)** 검증 통과한 문제만 사용자에게 제공한다.

### 2.3 코드 풀이 및 채점 `[MVP]`
- **FR-11 (MUST)** 사용자는 코드 에디터에서 코드를 작성·실행·제출할 수 있어야 한다.
- **FR-12 (MUST)** 제출 코드는 **Judge0 샌드박스**에서 hidden testcase로 채점되어야 한다.
- **FR-13 (MUST)** 채점 결과는 `AC / WA / TLE / RE / MLE` 상태로 분류되어야 한다.
- **FR-14 (MUST)** 채점은 **비동기**로 처리되며(큐), 사용자는 결과를 폴링/실시간으로 받는다.

### 2.4 풀이 지원 (챗봇형 힌트 · 오답 분석) `[MVP]`

#### 2.4.1 챗봇형 힌트 상호작용
- **FR-15 (MUST)** 힌트는 **챗봇 방식**으로 제공한다. 사용자가 대화창에서 힌트를 **요청할 때마다** 응답하며,
  자동으로 힌트를 밀어주지 않는다.
- **FR-16 (MUST)** 힌트 응답은 **문제 생성 시 저장된 힌트(2.1 FR-5)를 RAG로 검색**하여 구성한다.
  RAG가 검색하는 힌트 후보는 **현재 허용된 단계 이하로만 제한**해야 한다.

#### 2.4.2 힌트 단계 제어 (초과 차단)
- **FR-17 (MUST)** 힌트는 1단계(방향) → 2단계(알고리즘) → 3단계(구현)로 **순차적으로만** 상승한다.
- **FR-18 (MUST)** **현재 단계를 넘어가는 힌트 요청을 차단**해야 한다.
  - 사용자별·문제별로 "현재 도달한 힌트 단계"를 상태로 관리한다.
  - RAG/LLM에는 **허용 단계 이하의 힌트만 컨텍스트로 전달**한다. (상위 단계 힌트는 검색·노출 자체를 막는다)
  - 사용자가 아직 열지 않은 상위 단계를 요구하면, 다음 단계로의 **명시적 승급 확인**을 거친 뒤에만 단계를 올린다.
  - 즉, 프롬프트 인젝션 등으로 "3단계 힌트 바로 줘"라고 해도 저장/전달 단계에서 물리적으로 차단되어야 한다.

#### 2.4.3 힌트 노출 범위 (구조만 제공, 핵심 코드 금지)
- **FR-19 (MUST)** 힌트는 **문제의 전체 정답(전문)이나 핵심 코드를 제공하지 않는다.**
  - 최대 노출 범위는 **구조/뼈대(skeleton)** 까지다. 예: 함수 시그니처, 반복문 골격, 자료구조 선택,
    처리 단계의 주석(`# 여기서 lower_bound 계산`) 수준.
  - **핵심 로직(정답을 결정짓는 부분)은 반드시 사용자가 직접 입력**하도록 비워 둔다.
  - 힌트 생성/서빙 시 핵심 코드가 포함되지 않았는지 **검증(필터)** 을 거쳐야 한다.
- **FR-20 (MUST)** 정답 코드(reference solution)는 기본 비공개다.
  명시 요청 + "정말 확인하시겠습니까?" 확인 절차를 거친 경우에만 공개한다. (힌트 채널로는 절대 노출 금지)

#### 2.4.4 오답 분석
- **FR-21 (MUST)** 채점 상태에 따라 분기 처리해야 한다.
  (정답→로그 저장 / 오답→오답분석+반례 / TLE→시간복잡도 분석 / RE→에러 분석)
- **FR-22 (MUST)** 오답 원인 분석 시 RAG로 관련 개념 문서를 검색해 근거로 사용해야 한다.
- **FR-23 (SHOULD)** 실패한 케이스를 드러내는 반례를 생성·제시할 수 있어야 한다. (반례도 정답 코드를 노출하지 않는다)

### 2.5 학습 로그 / 개인화
- **FR-24 (SHOULD)** 사용자별 약점 유형, **사용한 힌트 단계**, 해결 여부를 학습 로그로 저장한다.
- **FR-25 (MAY)** Graph RAG(Neo4j)로 사용자의 과거 오답 개념을 반영한 맞춤형 문제를 생성한다.
- **FR-26 (MAY)** 학습 이력 기반으로 다음 학습 방향(약점 개념)을 추천한다.

### 2.6 사용자 계정
- **FR-27 (MUST)** 학습 로그·코드 소유권·개인화·힌트 단계 상태 관리를 위해 회원가입/로그인이 있어야 한다.
- **FR-28 (MUST)** 사용자는 자신의 학습 이력·코드를 삭제할 수 있어야 한다. (개인정보)

### 2.7 코드 공유 (커뮤니티)
- **FR-29 (MUST)** AC를 받은 사용자는 자신의 풀이 코드를 공개로 공유할 수 있어야 한다.
- **FR-30 (MUST)** **공유 gating** — 특정 문제의 공유 코드는 그 문제를 **스스로 AC한 사용자에게만** 노출한다.
  (아직 못 푼 사용자에게는 숨김 → 정답 유출/부정행위 방지)
- **FR-31 (SHOULD)** 커뮤니티 피드에서 문제별/알고리즘별/난이도별/인기순 필터를 제공한다.
- **FR-32 (SHOULD)** 공유 코드에 좋아요·댓글 기능을 제공한다.
- **FR-33 (MAY)** 같은 문제의 여러 풀이(접근법)를 비교해 볼 수 있어야 한다.

### 2.8 문제 신고 (Human-in-the-loop)
- **FR-34 (SHOULD)** 사용자는 품질 낮은 생성 문제를 신고할 수 있어야 한다.

## 3. 비기능 요구사항 (Non-Functional)

### 3.1 보안
- **NFR-1 (MUST)** 사용자 코드는 Judge0 샌드박스에서만 실행하고, timeout·memory·network를 제한한다.
- **NFR-2 (MUST)** Judge0 워커는 API·DB와 네트워크 격리하며 최소 권한으로 실행한다.
- **NFR-3 (MUST)** API 키·시크릿은 코드에 하드코딩하지 않고 환경변수(`.env`)로 관리한다.
- **NFR-4 (MUST)** 힌트 단계 제어(FR-18)와 핵심 코드 필터(FR-19)는 **서버 측에서 강제**한다.
  클라이언트가 단계나 노출 범위를 임의로 바꿀 수 없어야 한다.

### 3.2 신뢰성 / 품질
- **NFR-5 (MUST)** 생성 문제는 Validator 통과 없이는 사용자에게 제공하지 않는다.
- **NFR-6 (SHOULD)** LLM 출력(문제·힌트·피드백)은 Pydantic/Structured Output으로 스키마를 강제한다.
- **NFR-7 (SHOULD)** 잘못된 개념 설명(환각) 방지를 위해 RAG 출처 문서를 근거로 답한다.

### 3.3 성능
- **NFR-8 (SHOULD)** 힌트는 사전 생성·저장 후 RAG 서빙하므로, 힌트 응답은 LLM 즉석 생성보다 빠르고 저렴해야 한다.
- **NFR-9 (SHOULD)** 문제 생성·피드백 응답 시간은 학습 흐름을 방해하지 않는 수준이어야 한다.

### 3.4 확장성 / 유지보수
- **NFR-10 (MUST)** `packages/agent`는 웹/DB에 의존하지 않는 독립 패키지로 유지한다.
- **NFR-11 (SHOULD)** Vector DB / LLM Provider는 교체 가능하도록 추상화한다.

## 4. 힌트 데이터 흐름 (요약)

```
[문제 생성 시]
Problem Generator ── 힌트 1~3단계 생성 ──> [핵심코드 필터 검증] ──> DB 저장 + 힌트 벡터 스토어 색인

[풀이 중 사용자가 챗봇에 힌트 요청]
사용자 요청
  └─> 서버가 (사용자, 문제)의 "현재 허용 단계" 확인
        └─> RAG 검색 범위를 [1 .. 허용단계]로 제한
              └─> 검색된 힌트로 응답 구성 (구조까지만, 핵심 코드 제외)
        └─> 다음 단계 요청 시 → 승급 확인 → 허용 단계 +1 (상위 단계 힌트 노출 해제)
```

핵심 규칙 3가지:
1. **사전 생성·저장** — 힌트는 문제 생성 시 미리 만들어 DB에 넣고 RAG로 서빙한다. (FR-5, FR-16)
2. **단계 초과 차단** — 허용 단계보다 높은 힌트는 검색·전달 자체를 막는다. (FR-18, NFR-4)
3. **구조까지만** — 전문/핵심 코드는 절대 주지 않고 뼈대만 제공, 핵심 로직은 사용자가 입력. (FR-19)

## 5. 평가 지표 (기획 16장 요약)

| 구분 | 지표 |
|---|---|
| 문제 생성 품질 | 문제 유효성, 예제 정확도, 난이도 적합성, 알고리즘 적합성, 테스트케이스 다양성, 표절 위험도 |
| 풀이 지원 품질 | 힌트 유용성, **정답 노출률(낮을수록 좋음)**, 오답 원인 탐지율, 반례 적합성, RAG 검색 정확도, 학습 향상도 |
| 시스템 성능 | 평균 문제 생성 시간, 평균 피드백 응답 시간, 코드 실행 실패율, Validator 통과율, 사용자 만족도 |
